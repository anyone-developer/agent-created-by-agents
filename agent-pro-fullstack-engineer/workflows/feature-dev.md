# workflows/feature-dev.md — 功能开发流程

> 好的功能开发 = 清晰的需求 + 合理的设计 + 高质量的代码 + 充分的测试

---

## 🎯 功能开发全流程

```
需求理解 → 方案设计 → 任务拆解 → 编码实现 → 自测验证 → Code Review → 集成测试 → 上线部署
```

---

## 📋 Phase 1: 需求理解

### 输入
- 产品需求文档（PRD）
- 用户故事
- 设计稿（Figma/Sketch）

### 检查清单

```markdown
## 需求确认

### 功能描述
- [ ] 这个功能要解决什么问题？
- [ ] 目标用户是谁？
- [ ] 用户如何使用这个功能？
- [ ] 成功的标准是什么？

### 边界确认
- [ ] 这个功能的范围是什么？
- [ ] 什么不在范围内？
- [ ] 和现有功能有什么关联？

### 验收标准
- [ ] 功能验收标准（AC）是什么？
- [ ] 性能要求是什么？
- [ ] 兼容性要求是什么？

### 疑问澄清
- [ ] 有任何不清楚的地方？
- [ ] 有需要确认的技术细节？
```

### 输出
- 需求理解文档
- 问题清单及答案

## 📋 Phase 2: 方案设计

### 技术方案设计

```markdown
## 技术方案

### 影响范围分析
这个功能会影响哪些模块？
- 前端：
- 后端：
- 数据库：
- 第三方服务：

### 接口设计
#### API接口
```typescript
// 请求
POST /api/xxx
{
  field1: string;
  field2: number;
}

// 响应
{
  code: number;
  data: { ... };
  message: string;
}
```

### 数据模型
```typescript
interface FeatureModel {
  id: string;
  // ...
}
```

### 前端组件设计
```
FeaturePage/
├── index.tsx           # 页面入口
├── FeatureForm.tsx     # 表单组件
├── FeatureList.tsx     # 列表组件
├── FeatureDetail.tsx   # 详情组件
└── hooks/
    └── useFeature.ts   # 业务逻辑Hook
```

### 关键决策
1. 为什么选择这个方案？
2. 有哪些备选方案？
3. 这个方案的风险是什么？

### 数据流
用户操作 → 前端组件 → API请求 → 后端处理 → 数据库 → 响应 → 前端更新
```

### 设计评审
- [ ] 和前端/后端同事对齐方案
- [ ] 确认接口设计
- [ ] 确认数据模型
- [ ] 识别潜在风险

## 📋 Phase 3: 任务拆解

### WBS（工作分解结构）

```markdown
## 任务分解

### 后端任务
- [ ] [BE-1] 创建数据库Migration - 0.5h
- [ ] [BE-2] 实现Repository层 - 1h
- [ ] [BE-3] 实现Service层 - 2h
- [ ] [BE-4] 实现Controller层 - 1h
- [ ] [BE-5] 编写单元测试 - 1h
- [ ] [BE-6] API文档更新 - 0.5h

### 前端任务
- [ ] [FE-1] 创建页面组件 - 1h
- [ ] [FE-2] 实现表单交互 - 2h
- [ ] [FE-3] 对接API - 1h
- [ ] [FE-4] 处理加载/错误状态 - 1h
- [ ] [FE-5] 样式和响应式 - 1h
- [ ] [FE-6] 编写测试 - 1h

### 总计
- 后端：6h
- 前端：7h
- 联调：2h
- Buffer：2h
- **总计：17h（约2-3天）**
```

### 依赖关系
```
[BE-1] → [BE-2] → [BE-3] → [BE-4] → [BE-5]
                                    ↓
[FE-1] → [FE-2] → [FE-3] ←→ 联调 → 测试
```

## 📋 Phase 4: 编码实现

### 开发规范

```markdown
## 编码规范

### Git 分支
```bash
# 创建功能分支
git checkout -b feature/xxx

# 或修复分支
git checkout -b fix/xxx
```

### 提交规范
```bash
# 格式
<type>(<scope>): <description>

# 示例
feat(auth): add login API
fix(ui): resolve button alignment issue
docs(api): update Swagger documentation
test(user): add unit tests for UserService
```

### 代码风格
- 遵循项目ESLint配置
- 使用Prettier格式化
- 命名清晰，注释必要

### 编码检查清单
- [ ] 变量/函数命名清晰
- [ ] 没有魔法数字/字符串
- [ ] 错误处理完善
- [ ] 边界情况考虑
- [ ] 没有console.log（使用logger）
- [ ] TypeScript类型完整
```

### 开发流程

```markdown
## 每日开发

### 开始开发
1. git pull 最新代码
2. 确认任务优先级
3. 开始编码

### 开发中
1. 小步提交（每完成一个小功能）
2. 写必要的测试
3. 自测功能

### 结束开发
1. 运行测试
2. 代码自查
3. 提交PR
```

## 📋 Phase 6: Code Review

### PR 模板

```markdown
## 变更说明

### 功能描述
[描述这个PR实现了什么功能]

### 变更内容
- [变更1]
- [变更2]

### 测试情况
- [ ] 单元测试通过
- [ ] 手动测试通过
- [ ] 兼容性测试通过

### 截图/录屏
[如果有UI变更，提供截图]

### 影响范围
- 前端：
- 后端：
- 数据库：

### 关联Issue
#xxx
```

### Review 检查清单

```markdown
## Review 要点

### 正确性
- [ ] 逻辑正确
- [ ] 边界处理
- [ ] 错误处理

### 可读性
- [ ] 命名清晰
- [ ] 代码简洁
- [ ] 注释必要

### 性能
- [ ] 无明显性能问题
- [ ] 数据库查询优化
- [ ] 前端渲染优化

### 安全
- [ ] 输入验证
- [ ] 权限检查
- [ ] 敏感数据处理

### 测试
- [ ] 测试覆盖充分
- [ ] 测试用例合理
```

### Review 流程
1. 提交PR，指定Reviewer
2. Reviewer在24小时内完成Review
3. 根据反馈修改代码
4. Reviewer approve后合并
