# workflows/new-project.md — 新项目启动流程

> 好的开始是成功的一半。

---

## 🚀 新项目启动流程

### 阶段概览

```
需求理解 → 技术选型 → 架构设计 → 环境搭建 → 核心开发 → 测试验证 → 部署上线
   ↓          ↓          ↓          ↓          ↓          ↓          ↓
  1-2天      1天       1-2天       1天       2-4周      3-5天      1-2天
```

---

## 📋 Phase 1: 需求理解 (1-2天)

### 目标
- 完全理解项目要解决什么问题
- 明确目标用户和使用场景
- 确定核心功能和优先级

### 检查清单

```markdown
## 需求文档

### 项目概述
- [ ] 项目名称：
- [ ] 项目目标：
- [ ] 目标用户：
- [ ] 核心价值：

### 功能需求
#### P0（必须有）
- [ ] 功能1：描述、用户故事、验收标准
- [ ] 功能2：...

#### P1（应该有）
- [ ] 功能3：...

#### P2（可以有）
- [ ] 功能4：...

### 非功能需求
- [ ] 性能要求：响应时间、并发量
- [ ] 安全要求：认证、授权、数据保护
- [ ] 可用性要求：SLA、容灾
- [ ] 兼容性要求：浏览器、设备

### 约束条件
- [ ] 时间约束：
- [ ] 预算约束：
- [ ] 技术约束：
- [ ] 人员约束：

### 成功指标
- [ ] 如何衡量项目成功？
- [ ] 关键指标是什么？
```

### 关键问题清单

```
必问问题：
1. 这个产品的核心价值是什么？
2. 目标用户是谁？他们有什么痛点？
3. MVP范围是什么？什么功能是必须的？
4. 预期的用户量和数据量是多少？
5. 有什么技术限制或偏好？
6. 项目时间线是什么？
7. 团队有哪些人？技术水平如何？
8. 现有系统需要集成吗？
9. 有什么合规要求？
10. 成功的标准是什么？
```

## 📋 Phase 2: 技术选型 (1天)

### 目标
- 根据需求选择合适的技术栈
- 确保团队能够驾驭所选技术
- 考虑长期维护性

### 选型决策树

```
项目类型？
├── Web应用
│   ├── 需要SEO？ → Next.js / Nuxt
│   └── 不需要SEO？ → React / Vue SPA
├── 移动应用
│   ├── 跨平台？ → React Native / Flutter
│   └── 原生体验？ → Swift / Kotlin
├── API服务
│   ├── 快速开发？ → FastAPI / Express
│   ├── 高性能？ → Go (Gin) / Fastify
│   └── 企业级？ → NestJS / Spring Boot
└── 全栈应用
    ├── JavaScript团队 → Next.js (全栈)
    └── Python团队 → Django / FastAPI + React
```

### 技术栈模板

```yaml
# 标准Web应用技术栈
frontend:
  framework: Next.js 14+ (App Router)
  language: TypeScript
  styling: Tailwind CSS
  state: Zustand / React Query
  ui: shadcn/ui

backend:
  framework: NestJS
  language: TypeScript
  orm: Prisma
  database: PostgreSQL
  cache: Redis

devops:
  version_control: Git + GitHub
  ci_cd: GitHub Actions
  hosting: Vercel (前端) + Railway (后端)
  monitoring: Sentry + Vercel Analytics
```

## 📋 Phase 3: 架构设计 (1-2天)

### 目标
- 设计系统整体架构
- 定义模块边界和接口
- 画出架构图

### 架构设计流程

```
1. 画系统上下文图（谁在用系统？系统和谁交互？）
2. 画容器图（前端、后端、数据库、缓存...）
3. 画组件图（每个容器内部的组件）
4. 定义API接口
5. 设计数据库Schema
6. 编写架构决策记录（ADR）
```

### 输出物

```markdown
## 架构设计文档

### 系统上下文图
[插入系统上下文图]

### 容器图
[插入容器图]

### API设计
#### RESTful API
- GET /api/resources
- POST /api/resources
- GET /api/resources/:id
- PUT /api/resources/:id
- DELETE /api/resources/:id

### 数据库设计
[插入ER图]

### 架构决策记录
- ADR-001: 选择xxx的理由
- ADR-002: ...
```

## 📋 Phase 5: 核心开发 (2-4周)

### 目标
- 按优先级实现核心功能
- 保持代码质量
- 持续集成

### 开发节奏

```markdown
## Sprint 规划（1周一个Sprint）

### Sprint 1: 基础功能
- [ ] 用户认证（注册/登录）
- [ ] 基础UI框架
- [ ] 核心数据模型
- [ ] 基础API

### Sprint 2: 核心功能
- [ ] 功能1完整实现
- [ ] 功能2完整实现
- [ ] 前后端联调

### Sprint 3: 完善功能
- [ ] 功能3完整实现
- [ ] 错误处理
- [ ] 边界情况处理

### Sprint 4: 优化和测试
- [ ] 性能优化
- [ ] 安全加固
- [ ] 完整测试
```

### 每日开发流程

```markdown
## 每日流程

### 早上
1. 检查任务看板
2. 确认今日目标
3. 开始开发

### 开发中
1. 小步提交（每完成一个小功能就提交）
2. 写测试（至少核心功能要有测试）
3. 自测（自己先验证功能正常）

### 结束前
1. 推送代码
2. 更新任务状态
3. 记录遇到的问题
```

---

## 📋 Phase 6: 测试验证 (3-5天)

### 目标
- 确保功能正确
- 发现并修复问题
- 达到上线质量标准

### 测试策略

```markdown
## 测试金字塔

### 单元测试（70%）
- 工具函数
- 业务逻辑
- 数据处理

### 集成测试（20%）
- API接口
- 数据库操作
- 服务间交互

### E2E测试（10%）
- 核心用户流程
- 关键业务场景
```

### 测试检查清单

```markdown
## 测试清单

### 功能测试
- [ ] 所有功能按预期工作
- [ ] 边界情况处理正确
- [ ] 错误提示清晰

### 兼容性测试
- [ ] Chrome/Firefox/Safari
- [ ] 桌面/平板/手机
- [ ] 不同操作系统

### 性能测试
- [ ] 首屏加载 < 3秒
- [ ] API响应 < 500ms
- [ ] 无明显卡顿

### 安全测试
- [ ] 输入验证
- [ ] 认证授权
- [ ] 敏感数据保护

### 用户测试
- [ ] 让真实用户试用
- [ ] 收集反馈
- [ ] 修复问题
```

---

## 📋 Phase 7: 部署上线 (1-2天)

### 目标
- 安全部署到生产环境
- 确保系统稳定运行
- 建立监控体系

### 部署清单

```markdown
## 部署清单

### 部署前
- [ ] 所有测试通过
- [ ] 代码已合并到main分支
- [ ] 已创建Git Tag
- [ ] 数据库备份完成
- [ ] 环境变量配置完成
- [ ] 域名和SSL配置完成

### 部署中
- [ ] 部署后端
- [ ] 运行数据库迁移
- [ ] 部署前端
- [ ] 清除CDN缓存

### 部署后
- [ ] 验证核心功能
- [ ] 检查监控和日志
- [ ] 通知团队
- [ ] 更新文档
```

### 监控配置

```markdown
## 监控体系

### 应用监控
- 错误追踪：Sentry
- 性能监控：Vercel Analytics / Datadog
- 可用性监控：UptimeRobot

### 告警规则
- 错误率 > 1% → 告警
- 响应时间 > 3s → 告警
- 可用性 < 99.9% → 告警
```
