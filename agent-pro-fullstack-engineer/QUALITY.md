## 🧠 质量思维核心理念

### 1. 预防优于检测

```
错误的思维：先写代码，再写测试，最后修 bug
正确的思维：写代码时就想好怎么测，边写边测

质量成本曲线：
修复成本 = 设计阶段 × 1
         = 编码阶段 × 5
         = 测试阶段 × 10
         = 上线之后 × 100
```

### 2. 质量是每个人的责任

```
不是"QA 会测的" → 而是"我先保证质量"
不是"上线后监控" → 而是"上线前预防"
不是"用户反馈 bug" → 而是"我们先发现并修复"
```

### 3. 恰到好处的质量投入

```
80/20 法则：
- 20% 的代码有 80% 的 bug
- 核心业务逻辑 → 高质量标准
- 辅助功能 → 适度质量标准
- 实验性功能 → 快速验证优先

质量投入决策矩阵：
           高影响          低影响
高风险   │ 必须完美    │ 充分测试
         │ (支付模块)   │ (管理后台)
低风险   │ 充分测试    │ 基本验证
         │ (用户注册)   │ (彩蛋功能)
```

## 📐 代码质量标准

### 命名规范

```markdown
## 命名黄金法则

### 变量命名
✅ 好的命名：
- `userEmail` — 清晰表达含义
- `isAuthenticated` — 布尔值用 is/has/can 前缀
- `retryCount` — 名词表达数量

❌ 差的命名：
- `data` — 太模糊
- `temp` — 无意义
- `x` — 只在数学公式中可接受
- `userInfo2` — 数字后缀表示设计有问题

### 函数命名
✅ 好的命名：
- `getUserById()` — 动词开头，清晰表达行为
- `calculateTotalPrice()` — 表达计算逻辑
- `validateEmailFormat()` — 表达验证意图

❌ 差的命名：
- `process()` — 处理什么？
- `handleData()` — 怎么处理？
- `doStuff()` — 完全无意义

### 类/组件命名
✅ 好的命名：
- `UserProfile` — 名词，表达实体
- `PaymentProcessor` — 表达职责
- `ShoppingCart` — 业务概念清晰

### 常量命名
✅ 好的命名：
- `MAX_RETRY_COUNT = 3`
- `API_BASE_URL = "https://api.example.com"`
- `DEFAULT_TIMEOUT_MS = 5000`
```

### 函数设计原则

```markdown
## 函数质量标准

### 单一职责
✅ 每个函数只做一件事：
```javascript
// 好：职责单一
function calculateDiscount(price, discountRate) {
  return price * discountRate;
}

function formatPrice(amount) {
  return `$${amount.toFixed(2)}`;
}

// 差：职责混杂
function processOrder(order) {
  // 计算价格 + 格式化 + 发送邮件 + 更新数据库...
}
```

### 函数长度
- 理想：10-20 行
- 上限：50 行（超过就需要拆分）
- 例外：复杂的 switch/case 或数据映射可以略长

### 参数数量
- 理想：0-3 个参数
- 上限：5 个参数
- 超过 5 个 → 封装为对象

### 复杂度控制
- 圈复杂度 < 10
- 嵌套层级 < 3
- 如果复杂度高 → 提取子函数
```

### 注释规范

```markdown
## 注释原则

### 好注释：解释 WHY，不是 WHAT
✅ 好：
```javascript
// 使用二分查找是因为数据已排序，可以将 O(n) 降为 O(log n)
function findUser(sortedUsers, targetId) {
  // ...
}
```

❌ 差：
```javascript
// 循环遍历数组
for (let i = 0; i < arr.length; i++) {
  // ...
}
```

### 必须注释的场景
1. **复杂算法** — 解释思路和时间/空间复杂度
2. **非直观的业务逻辑** — 为什么要这样做
3. **临时解决方案** — TODO/FIXME/HACK 标记
4. **外部依赖的坑** — 已知的限制或注意事项
5. **正则表达式** — 解释匹配规则

### JSDoc / TSDoc
```typescript
/**
 * 计算用户订单的最终价格
 * 
 * @param order - 订单对象，包含商品列表
 * @param coupon - 优惠券代码，可选
 * @param shippingMethod - 配送方式
 * @returns 包含小计、折扣、运费、总价的价格明细
 * @throws {InvalidCouponError} 当优惠券无效时抛出
 * 
 * @example
 * const pricing = calculatePricing(order, 'SAVE10', 'express');
 * console.log(pricing.total); // 89.99
 */
```

### 反模式注释
❌ 禁止：
- 注释掉的代码（用 Git 管理版本，不要注释）
- 无意义的注释（`// i++ // 递增 i`）
- 过时的注释（与代码不一致）
```

## 🏗️ 架构质量标准

### 模块化质量

```markdown
## 模块质量检查清单

### 高内聚
- [ ] 模块内的功能都服务于同一目标？
- [ ] 模块的职责可以一句话说清？
- [ ] 改动一个功能很少需要改其他文件？

### 低耦合
- [ ] 模块之间通过清晰的接口通信？
- [ ] 改一个模块不会导致其他模块崩溃？
- [ ] 可以独立测试每个模块？

### 依赖方向
- [ ] 依赖关系是单向的？（A → B → C，不要循环依赖）
- [ ] 高层模块不依赖低层模块的具体实现？
- [ ] 使用依赖注入而非硬编码？
```

### API 设计质量

```markdown
## API 质量标准

### RESTful 设计
✅ 好的 API：
```
GET    /api/users          # 获取用户列表
GET    /api/users/:id      # 获取单个用户
POST   /api/users          # 创建用户
PUT    /api/users/:id      # 完整更新用户
PATCH  /api/users/:id      # 部分更新用户
DELETE /api/users/:id      # 删除用户
```

❌ 差的 API：
```
GET /api/getUsers
POST /api/createUser
POST /api/deleteUser       # 用 POST 删除
GET  /api/updateUser       # 用 GET 更新
```

### 响应格式一致性
```json
// 成功响应
{
  "success": true,
  "data": { ... },
  "meta": {
    "page": 1,
    "total": 100
  }
}

// 错误响应
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "邮箱格式不正确",
    "details": [...]
  }
}
```

### 版本管理
- URL 版本：`/api/v1/users`
- Header 版本：`Accept: application/vnd.api.v1+json`
- 必须有版本策略！
```

## 🧪 测试质量标准

### 测试覆盖率目标

```markdown
## 覆盖率要求

### 核心业务逻辑
- 行覆盖率：≥ 80%
- 分支覆盖率：≥ 75%
- 关键路径：100%

### 工具函数/公共库
- 行覆盖率：≥ 90%
- 边界条件：必须覆盖

### UI 组件
- 核心交互流程：必须有 E2E 测试
- 组件渲染：至少有快照测试
- 可选：组件单元测试

### API 端点
- 每个端点至少一个集成测试
- 错误场景必须测试
- 认证/授权必须测试
```

### 测试质量标准

```markdown
## 测试本身的质量

### 好的测试特征
1. **独立性** — 测试之间不依赖执行顺序
2. **确定性** — 同样的输入总是同样的结果
3. **快速** — 单元测试 < 10ms，集成测试 < 1s
4. **清晰** — 测试名即文档：`it('should reject invalid email format')`
5. **单一断言** — 每个测试主要验证一件事

### 测试命名规范
```javascript
// 模式：should [expected behavior] when [condition]

describe('UserService', () => {
  describe('createUser', () => {
    it('should return user with id when valid data provided', () => {});
    it('should throw ValidationError when email is invalid', () => {});
    it('should throw DuplicateError when email already exists', () => {});
  });
});
```

### 测试反模式
❌ 避免：
- 测试实现细节（应该测试行为，不是实现）
- 过度 mock（mock 太多说明耦合太紧）
- 模糊的断言（`expect(result).toBeTruthy()` 不如 `expect(result.id).toBe(123)`）
- 测试无意义的东西（测试框架本身的行为）
```

### 何时使用 Hotfix

### Hotfix 质量标准
