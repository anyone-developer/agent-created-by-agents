# skills/architecture.md — 系统架构设计能力

> 好的架构不是设计出来的，是演化出来的。但好的起点能让演化更顺畅。

---

## 🏗️ 架构思维

### 核心理念

作为 FullStack，我理解架构的本质是**权衡**：

- **没有完美的架构**，只有适合当前场景的架构
- **架构是演化的**，不是一成不变的
- **过度设计是毒药**，设计不足是慢性病
- **简单性是终极的复杂**——能简单解决的问题不要复杂化

### 架构设计原则

```
1. 适度设计 — 满足当前需求，预留扩展空间
2. 关注点分离 — 每个模块只做一件事
3. 低耦合高内聚 — 模块间依赖最小化
4. 可测试性 — 架构要便于测试
5. 可观测性 — 系统运行状态要透明
```

## 📐 架构设计流程

### Step 1: 需求理解

```
问自己：
- 这个系统要解决什么问题？
- 用户是谁？他们怎么使用？
- 核心功能是什么？辅助功能是什么？
- 性能要求？并发量？数据量？
- 团队规模和技术栈？
- 时间和预算约束？
```

### Step 2: 领域分析

```markdown
## 领域模型
1. 识别核心领域概念
2. 定义实体（Entity）和值对象（Value Object）
3. 确定聚合根（Aggregate Root）
4. 定义领域服务（Domain Service）
5. 识别限界上下文（Bounded Context）

## 示例：电商系统
- 核心域：商品、订单、支付
- 支撑域：用户、库存、物流
- 通用域：通知、日志、权限
```

### Step 3: 架构风格选择

| 架构风格 | 适用场景 | 优点 | 缺点 |
|---------|---------|------|------|
| **单体架构** | 小型项目、MVP、团队<5人 | 简单、部署容易、调试方便 | 扩展性差、技术栈锁定 |
| **分层架构** | 中型项目、业务逻辑复杂 | 关注点分离、易于维护 | 性能损耗、可能过度分层 |
| **微服务** | 大型项目、多团队、高并发 | 独立部署、技术栈灵活 | 分布式复杂、运维成本高 |
| **事件驱动** | 异步处理、实时系统 | 解耦、可扩展 | 最终一致性、调试困难 |
| **CQRS** | 读写分离、复杂查询 | 性能优化、灵活扩展 | 复杂度高、数据同步 |
| **Serverless** | 流量波动大、事件触发型 | 按需付费、免运维 | 冷启动、厂商锁定 |

### Step 4: 技术选型

```
选型考虑因素：
1. 团队熟悉度 — 团队会用什么？
2. 社区活跃度 — 遇到问题能查到资料吗？
3. 长期维护 — 这个技术会持续更新吗？
4. 生态丰富度 — 有我们需要的库和工具吗？
5. 性能表现 — 满足我们的性能要求吗？
6. 招聘难度 — 能招到会用这个技术的人吗？
```

### Step 5: 架构图绘制

```markdown
## C4模型（推荐）

### Level 1: 系统上下文图（System Context）
- 展示系统与外部用户、系统的关系
- 适合：向非技术人员解释系统

### Level 2: 容器图（Container）
- 展示系统的技术容器（应用、数据库、消息队列等）
- 适合：向技术人员解释技术架构

### Level 3: 组件图（Component）
- 展示容器内部的组件
- 适合：开发团队内部沟通

### Level 4: 代码图（Code）
- 展示组件内部的类/模块
- 适合：详细设计文档
```

## 🎯 常见架构模式

### 前端架构

```markdown
## 组件化架构
```

```
src/
├── components/          # 可复用组件
│   ├── ui/             # 基础UI组件（Button、Input、Modal）
│   ├── layout/         # 布局组件（Header、Footer、Sidebar）
│   └── business/       # 业务组件
├── pages/              # 页面组件
├── hooks/              # 自定义Hooks
├── services/           # API服务
├── stores/             # 状态管理
├── utils/              # 工具函数
├── types/              # TypeScript类型定义
└── styles/             # 全局样式
```

```markdown
## 状态管理策略

| 状态类型 | 存储位置 | 示例 |
|---------|---------|------|
| 本地状态 | useState/useReducer | 表单输入、UI状态 |
| 全局状态 | Zustand/Redux | 用户信息、主题 |
| 服务端状态 | React Query/SWR | API数据 |
| URL状态 | URL参数 | 筛选条件、分页 |
| 持久化状态 | localStorage | 用户偏好 |
```

### 后端架构

```markdown
## 分层架构（推荐）
```

```
src/
├── controllers/        # 控制层 - 处理HTTP请求
├── services/          # 业务层 - 业务逻辑
├── repositories/      # 数据层 - 数据访问
├── models/            # 数据模型
├── dto/               # 数据传输对象
├── middlewares/       # 中间件
├── utils/             # 工具函数
├── config/            # 配置
└── exceptions/        # 异常处理
```

```markdown
## 依赖注入（DI）
```

```typescript
// NestJS 示例
@Injectable()
export class HotelService {
  constructor(
    private readonly hotelRepository: HotelRepository,
    private readonly bookingService: BookingService,
    private readonly notificationService: NotificationService,
  ) {}

  async findById(id: string): Promise<Hotel> {
    return this.hotelRepository.findById(id);
  }
}
```

### 数据库架构

```markdown
## 数据库选型决策树
```

```
需要事务支持？
├── 是 → 关系型数据库
│   ├── 简单查询多 → PostgreSQL
│   └── 读多写少 → MySQL + 读写分离
└── 否 → 
    ├── 数据结构灵活 → MongoDB
    ├── 需要缓存 → Redis
    ├── 需要全文搜索 → Elasticsearch
    └── 时序数据 → InfluxDB / TimescaleDB
```

## 📊 架构质量属性

### 性能

```markdown
## 性能优化策略

1. **缓存策略**
   - 浏览器缓存（Cache-Control）
   - CDN缓存（静态资源）
   - 应用缓存（Redis）
   - 数据库缓存（查询缓存）

2. **数据库优化**
   - 索引优化
   - 查询优化（避免N+1）
   - 读写分离
   - 分库分表

3. **前端优化**
   - 代码分割（Code Splitting）
   - 懒加载（Lazy Loading）
   - 图片优化（WebP、响应式图片）
   - 预加载（Prefetch）
```

### 可扩展性

```markdown
## 扩展策略

1. **水平扩展**
   - 无状态服务（易扩展）
   - 负载均衡
   - 数据库分片

2. **垂直扩展**
   - 升级硬件
   - 优化代码

3. **功能扩展**
   - 插件化架构
   - 微服务拆分
   - 事件驱动
```

### 可维护性

```markdown
## 维护性保障

1. **代码规范**
   - 统一的代码风格
   - ESLint/Prettier
   - 代码审查

2. **文档完善**
   - 架构决策记录（ADR）
   - API文档
   - 部署文档

3. **测试覆盖**
   - 单元测试
   - 集成测试
   - E2E测试

4. **监控告警**
   - 应用监控
   - 错误追踪
   - 性能监控
```

### 可用性

```markdown
## 高可用策略

1. **冗余设计**
   - 多实例部署
   - 数据库主从
   - 异地多活

2. **故障恢复**
   - 健康检查
   - 自动重启
   - 故障转移

3. **降级策略**
   - 服务降级
   - 熔断机制
   - 限流保护
```

## 🚨 架构反模式

### 避免的陷阱

```markdown
❌ **过度工程**
- 症状：设计了用不到的复杂架构
- 解决：YAGNI原则，先简单后复杂

❌ **简历驱动开发**
- 症状：为了用新技术而用新技术
- 解决：根据实际需求选型

❌ **分布式单体**
- 症状：微服务但紧耦合
- 解决：明确服务边界，异步通信

❌ **数据库即API**
- 症状：直接暴露数据库给前端
- 解决：引入API层/GraphQL

❌ **上帝对象**
- 症状：一个类/模块做了所有事情
- 解决：单一职责原则
```

## 🎓 架构能力自检清单

- [ ] 我理解了业务需求吗？
- [ ] 我选择了合适的架构风格吗？
- [ ] 我考虑了非功能性需求（性能、安全、可扩展）吗？
- [ ] 我的技术选型有充分的理由吗？
- [ ] 我画了架构图吗？
- [ ] 我写了架构决策记录吗？
- [ ] 我和团队对齐了架构方案吗？
- [ ] 我考虑了演进路径吗？
