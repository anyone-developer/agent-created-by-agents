## 📝 记忆管理规则

### 何时记录

- ✅ 做了重要技术决策时
- ✅ 遇到并解决了复杂问题时
- ✅ 学到了新的最佳实践时
- ✅ 发现了某个反模式时
- ✅ 用户反馈了偏好时

### 记录格式

- **简洁**：能用一行说清的不用两行
- **可搜索**：使用明确的关键词
- **有上下文**：记录为什么，不只是是什么

### 定期回顾

- 每周回顾一次经验教训
- 每个项目结束后总结
- 更新技术栈偏好（技术在演进）

## 🚀 运维知识库 (DevOps Memory)

作为全栈工程师，运维经验是重要的长期记忆。

### 常见故障与解决方案

```markdown
## 数据库连接池耗尽
- 症状：应用响应慢，数据库连接数达到上限
- 原因：连接未正确释放，或并发突增
- 解决：检查连接释放逻辑，调整池大小，添加连接超时

## 内存泄漏
- 症状：应用内存持续增长，最终 OOM
- 原因：未清理的定时器、闭包引用、事件监听器
- 解决：使用 heapdump 分析，检查全局变量，添加内存监控

## 磁盘空间耗尽
- 症状：写入失败，服务异常
- 原因：日志未轮转、临时文件堆积、备份未清理
- 解决：配置日志轮转，定期清理临时文件，监控磁盘使用

## DNS 解析失败
- 症状：服务间调用失败
- 原因：DNS 缓存过期、DNS 服务器故障
- 解决：添加 DNS 缓存，配置备用 DNS，使用 IP 直连作为 fallback
```

### 性能基准

```markdown
## 响应时间目标
- API 响应：< 200ms (P95)
- 页面加载：< 2s (首屏)
- 数据库查询：< 50ms (简单查询)

## 可用性目标
- 核心服务：99.9% (每月停机 < 43 分钟)
- 非核心服务：99.5% (每月停机 < 3.6 小时)

## 容量规划
- CPU：日常 < 60%，峰值 < 80%
- 内存：日常 < 70%，峰值 < 85%
- 磁盘：日常 < 70%，告警线 80%
```

### 运维工具速查

```markdown
## 常用命令
- 日志查看：`kubectl logs -f pod-name`
- 进入容器：`kubectl exec -it pod-name -- /bin/sh`
- 资源监控：`kubectl top pods`
- 端口转发：`kubectl port-forward pod-name 8080:80`

## 排查思路
1. 检查服务状态（是否运行中）
2. 查看日志（最近的错误）
3. 检查资源（CPU/内存/磁盘）
4. 检查网络（连接、DNS）
5. 检查依赖（数据库、缓存、第三方服务）
```
